Memintel
Internal Platform API Reference
Complete reference for SDK authors, platform teams, and CI/CD pipelines
Overview
The Memintel Internal Platform API is the full engine surface — a superset of the App Developer API. It exposes the compiler layer, full registry governance, semantic tooling, the LLM refinement loop, and all lifecycle management. Where the App Developer API is a product, this API is the engine itself.
Who this API is for:
SDK authors building the Python SDK on the full platform surface
Platform teams compiling, versioning, and governing definitions
CI/CD pipelines batch-deploying definitions and running semantic checks
Internal services producing meaning consumed by app developers
API Structure
All execution, condition evaluation, actions, definition authoring, registry discovery, intelligence, and AI-assisted authoring endpoints — everything available to app developers.
Compiler internals, full registry governance, advanced agent workflows, and legacy deprecated intelligence endpoints. Not exposed to app developers.
Execution Pipeline
Concept (ψ)  →  /execute              →  Result (Rₜ)
Condition (φ) →  /evaluate/condition  →  DecisionResult (Aₜ)
Action (α)    →  /actions/{id}/trigger →  ActionResult
All three in one call:  POST /evaluate/full
Definitions
Definition endpoints create the semantic building blocks of Memintel. A Concept declares meaning (ψ). A Condition declares significance (φ). A Primitive is an interface to a raw data source. All share the namespace.id versioning model.
Create concept
Registers a Concept definition — a semantic declaration of what something means for an entity. Concepts are authoring-time objects: you define them once, the compiler produces an execution graph, and the runtime evaluates them.
Response codes
Create primitive
Registers a Primitive definition — an interface to a raw data source. Primitives declare what data exists and where it comes from. They do NOT perform computation; computation belongs to Concepts.
Response codes
Create condition
Registers a Condition definition. Conditions interpret concept results: they declare a strategy (threshold, percentile, z_score, change, equals, or composite), a reference frame, and bind to a specific concept version.
Key request fields
Response codes
Create definitions (batch)
Submits multiple concept and/or primitive definitions for creation in a single request. Designed for CI/CD pipeline deploys, SDK sync operations, registry migrations, and bulk imports. Each definition is validated independently — partial success is possible. Definitions within the batch are processed in dependency order.

Execution
All execution is deterministic when a timestamp is provided. POST /execute/graph is an Internal Platform addition that bypasses compilation for maximum throughput. All other execution endpoints are available to app developers.
Execute concept
Evaluates a compiled concept for a given entity. With timestamp: deterministic and reproducible. Without timestamp: snapshot mode (current data, not guaranteed reproducible). Supports inline explanation, cache bypass, and version policy override.
Key request fields
Response — Result (Rₜ)
Response codes
Execute using precompiled graph
Executes a previously compiled ExecutionGraph directly by graph_id, bypassing the compilation step entirely. This is the highest-performance execution path — compilation is paid once at deploy time and amortised across every call. Optionally supply ir_hash to verify cache consistency and produce an audit trail.
Key request fields
Response codes
Evaluate condition
Evaluates a Condition (φ) against a concept's Result (Rₜ) to produce a DecisionResult (Aₜ). If the concept result is not already cached, the runtime implicitly executes the concept first — transparent but has latency implications on first call. Pre-warm with POST /execute if latency matters.
Key request fields
Response — DecisionResult (Aₜ)
Response codes
Evaluate condition (batch)
Evaluates a single condition across multiple entities in parallel. Returns an array of DecisionResult objects, one per entity. Partial failure is possible — always inspect each item's status.
Key request fields
Response codes
Execute full pipeline
Runs concept execution (ψ), condition evaluation (φ), and action triggering (α) in one atomic request — the recommended default for most production integrations. Returns a FullPipelineResult with result (Rₜ), decision (Aₜ), and action outcomes.

Compilation
The compiler transforms semantic definitions into execution graphs and semantic views. These endpoints are not exposed in the App Developer API. Every concept must be compiled before it can be executed via POST /execute/graph.
Compile definition
Compiles a semantic Concept definition into a fully resolved, typed, immutable ExecutionGraph (IR). The returned graph_id is stable for a given definition version. Cache graph_id at service startup and pass it to POST /execute/graph for maximum throughput.

Response — ExecutionGraph
Response codes
Compile semantic view
Returns the semantic view of a compiled concept — the meaning layer, not the execution layer. Exposes resolved features with semantic types and meaning_hash, the normalised canonical form, the dependency graph at the meaning level, and any detected semantic equivalences. Use for deduplication checks and as input to the semantic-refine loop.

Registry
The registry is the source of truth for all Memintel definitions. App developer access is read-only. Internal teams have full write, lifecycle management, and semantic analysis access.
Register definition
Stores a validated definition in the registry under its namespace. Definitions are immutable once registered — updates require a new version. Returns 409 if the same id + version already exists.
Request body (RegisterRequest)
Response codes
List definitions
Returns a paginated list of definitions in the registry. Filter by type, namespace, or tags. For free-text semantic search, use GET /registry/search instead.

Feature Registry
Features are first-class semantic primitives — registered independently of any Concept and reusable across many Concepts. The feature registry is the foundational layer of the Memintel meaning graph. Internal teams register, deduplicate, and analyse feature usage.
Register feature
Registers a Feature as a first-class entity. On registration, the compiler computes the meaning_hash and scans for duplicates. The on_duplicate policy controls what happens on a collision.

Actions
Actions are the α layer — the response layer that fires when a Condition evaluates. Actions are declarative: they define WHAT to trigger and HOW, not WHEN (the Condition's trigger.fire_on governs WHEN). Execution types: webhook, notification, workflow, register.
Create action
Defines an action (α) — the response layer of the Memintel pipeline. Each Action is bound to a specific Condition version for reproducibility via trigger.condition_id and trigger.condition_version.

Intelligence
Intelligence endpoints operate on the significance layer (DecisionResult Aₜ), not on raw concept outputs. The primary endpoint is POST /intelligence/condition-impact. The six legacy endpoints are deprecated and superseded by it.
Condition impact analysis
Condition-centric analytics. Analyses how frequently and under what circumstances a Condition fires — operating on DecisionResult (Aₜ), the significance layer, not raw concept outputs. Use for monitoring decision rate trends, detecting decision drift, governance audits, and action calibration.

Deprecated intelligence endpoints

Agents
Natural language query
Accepts a natural language query, resolves it to one or more concept evaluations, and returns structured results. The resolved_concepts field exposes which concepts were selected, enabling verification or override.
Key request fields
Response — AgentQueryResponse
Response codes
Generate definition
Converts a natural language description into a valid MemSDL definition via LLM-assisted generation. The generated definition is validated before being returned but is NOT registered or executed automatically. You must call POST /definitions/validate, then POST /registry/definitions, then POST /compile.
Key request fields
Response — AgentDefineResponse
Response codes
Generate condition
Generates a Condition definition from natural language. Same non-determinism notice and authoring workflow as POST /agents/define applies.
Key request fields
Response codes
Semantic refinement loop
Closes the LLM generation loop. Takes an existing Concept definition, its SemanticGraph from POST /compile/semantic, and a refinement instruction. Returns a revised definition with an updated semantic view and a diff showing exactly what changed in meaning. This is the native pattern for iterative concept authoring with compiler feedback.
Refinement loop (7 steps)
1. Generate initial definition  POST /agents/define
2. Compile semantic view  POST /compile/semantic
3. Refine  POST /agents/semantic-refine → revised definition + updated SemanticGraph
4. Repeat steps 2–3 until satisfied
5. Validate  POST /definitions/validate
6. Compile  POST /compile → ExecutionGraph
7. Register  POST /registry/definitions
Key request fields
Response — SemanticRefineResponse
Response codes
Compile workflow
Converts a natural language workflow description into an explicit, static, MemSDL-backed ExecutionPlan. No dynamic reasoning occurs at execution time — all steps are pre-resolved and validated before any execution begins. The SDK's agent.run() method calls this internally before executing a workflow.
